# 中文优先视频工作台设计

## 1. 目标
当前 `easy-manim` 已经具备可用的人类操作台，支持登录、创建任务、查看结果、管理记忆、画像和评测；但它的整体体验仍然更接近“工程控制台”，而不是“面向中文内容用户的视频工作台”。

本设计的目标是在不推翻现有任务驱动内核的前提下，把 Web UI 明确调整为一套更贴近真实中文用户心智的产品界面：

1. 以中文为默认语言，而不是英文界面的本地化补丁。
2. 首页优先展示最近生成的视频，而不是优先展示任务 ID 和状态列表。
3. 已完成任务必须拥有贴近内容的展示名称，而不是仅展示无意义的内部 ID。
4. 任务详情页要围绕“看结果、判断问题、继续改”展开，而不是围绕底层任务字段展开。
5. 记忆、画像、评测保留为重要能力页，但不抢占主创作入口。

## 2. 非目标
第一版设计明确不追求以下能力：

1. 完整的视频素材管理系统。
2. 面向多租户团队的权限与协作系统。
3. 复杂的视频剪辑时间线编辑器。
4. 完整的作品发布、分享、评论流程。
5. 所有后端错误的语义级自动修复。

第一版只解决“中文优先”“视频优先”“任务命名更贴近内容”这三个核心问题。

## 3. 关键设计判断
本次设计最重要的判断不是“把文案翻成中文”，而是：

**主用户真正关心的是最近做出了什么视频、是否满意、下一步怎么改，而不是系统内部任务编号。**

因此推荐把当前操作台从“任务列表优先”调整为“创作与回看优先”：

1. 首页首先服务于创作启动和最近视频回看。
2. 任务只是一条生产链路，不应压过视频结果本身。
3. 内部 `task_id` 仍然保留，但只能作为辅助信息存在。
4. 用户面对的主对象应该是“作品 / 视频 / 当前版本”，而不是“数据库中的任务记录”。

## 4. 产品定位
推荐把整套 UI 定义成：

**中文创作工作台 + 最近视频库 + 任务迭代页**

这个定位比“运维控制台”更适合当前目标用户，原因如下：

1. 中文用户打开页面后最希望先看到结果，而不是先学系统结构。
2. 最近视频是最强的反馈信号，能直接驱动下一步修订。
3. 任务、记忆、画像、评测更适合作为优化层，而不是一进来就占满主视图。

## 5. 信息架构
推荐导航结构如下：

1. `任务`
   作为首页，负责创建任务、回看最近视频、进入最近任务。
2. `视频`
   作为独立浏览页，专门查看最近生成的视频作品。
3. `记忆`
   查看和管理会话记忆、长期记忆。
4. `画像`
   查看评分卡、应用补丁、处理建议。
5. `评测`
   查看评测运行和用例级结果。

这个结构中的关键点是：

1. `任务` 是创作入口。
2. `视频` 是作品回看入口。
3. `记忆 / 画像 / 评测` 是质量优化入口。

## 6. 首页设计：创作台
首页应被正式定义为“创作台”，而不是普通任务列表页。

### 6.1 首页目标
首页只做三件事：

1. 快速发起一个新任务。
2. 立即看到最近生成的视频。
3. 继续进入某个任务做修订。

### 6.2 桌面端布局
推荐首屏采用双主轴布局：

1. 左侧：中文任务输入区
   1. 大文本框
   2. 中文快捷示例
   3. 创建按钮
   4. 轻量输入建议
2. 右侧：最近成功视频大播放器
   1. 直接播放最近完成的视频
   2. 展示任务名称
   3. 展示简短摘要
   4. 提供“继续修订”“查看详情”“打开视频”动作

首屏下方再分成两个区域：

1. 左侧：最近视频流
2. 右侧：最近任务队列

### 6.3 首页交互原则
首页应遵循以下交互原则：

1. 用户不必点进详情页才能开始回看视频。
2. 用户不必先理解 `task_id` 才能知道哪个结果值得继续改。
3. 最近视频区比最近任务区更强调视觉结果。
4. 首页的文案全部使用中文，不出现“Prompt”“Create task”“Refresh”这类混杂英文词。

## 7. 视频页设计：最近视频库
推荐新增独立的 `视频` 页面，而不是继续把视频回看强行塞在任务列表里。

### 7.1 页面目标
视频页专门解决“浏览最近作品”和“快速回到迭代链路”。

### 7.2 页面结构
推荐结构如下：

1. 顶部筛选栏
   1. 状态筛选：`全部 / 已完成 / 失败`
   2. 时间筛选：最近 24 小时、7 天、30 天
   3. 关键词搜索：按任务名称或摘要检索
2. 视频卡片流
   1. 任务名称
   2. 状态
   3. 更新时间
   4. 摘要
   5. 预览图或视频缩略播放
   6. `继续修订`
   7. `查看详情`

### 7.3 卡片内容优先级
每张视频卡片的信息层级固定为：

1. `display_title`
2. 状态
3. 简短摘要
4. 更新时间
5. `task_id`（弱化展示）

内部 ID 只允许放在次级信息区域，不能作为主标题。

## 8. 任务详情设计：视频工作台
任务详情页应从“任务字段详情页”升级为“视频工作台”。

### 8.1 页面目标
任务详情页只围绕三个动作展开：

1. 看当前结果
2. 判断哪里有问题
3. 用中文继续修订

### 8.2 推荐布局
推荐采用左右双栏结构：

1. 左侧主区
   1. 大播放器
   2. 预览帧
   3. 版本胶片带
2. 右侧决策区
   1. 结果摘要
   2. 校验结论
   3. 问题提示
   4. 修订输入框
   5. 失败重试 / 取消任务 / 查看资源

### 8.3 顶部信息
页面顶部信息顺序推荐为：

1. 任务名称 `display_title`
2. 状态、阶段、更新时间
3. 内部 `task_id`

其中 `task_id` 只做次级信息，不应压过任务名称。

### 8.4 版本对比
推荐引入“结果对比”而不是复杂技术 diff：

1. 支持切换 `当前版本 / 上一版本`
2. 支持并排预览两个版本
3. 页面给出中文自然语言对比提示，例如：
   1. 字幕更清晰
   2. 节奏更慢
   3. 结尾停留仍偏短

### 8.5 修订交互
修订区应支持中文自然输入，同时提供轻量快捷入口：

1. 颜色风格
2. 字幕文案
3. 节奏快慢
4. 镜头结构
5. 画面信息量

这能显著降低用户从“看问题”到“继续改”的认知切换成本。

## 9. 任务命名设计
“完成的任务应该有一个贴近的名字，而不是无意义的 ID”已经被确认是设计刚需。

### 9.1 命名目标
命名方案必须满足：

1. 用户能快速识别这个任务做的是什么。
2. 列表和视频页能自然浏览，不会出现一排 `task-17`、`task-18`。
3. 内部追踪仍可依赖 `task_id`，不破坏现有工作流。

### 9.2 推荐字段模型
建议把任务标识拆成两层：

1. `task_id`
   系统内部唯一标识，保留现有行为。
2. `display_title`
   前端主展示名称，用于列表、视频页、详情页标题。
3. `title_source`
   标记标题来源：
   1. `prompt`
   2. `result_refined`
   3. `manual`

### 9.3 生成规则
推荐采用“双轨命名”：

1. 创建任务时先从中文任务描述中提炼一个展示名。
2. 如果原始描述过长，取核心主题。
3. 如果原始描述过泛，在任务完成后结合结果摘要进行二次优化。
4. 用户允许手动修改展示名一次，修改后 `title_source = manual`。

### 9.4 中文标题规范
建议标题遵循以下规则：

1. 8 到 20 个中文字符优先。
2. 优先包含内容主体，而不是动作词堆叠。
3. 避免“任务一”“生成视频”“测试结果”这类空泛标题。
4. 优先类似：
   1. `蓝色圆形开场动画`
   2. `季度营收柱状图`
   3. `带中文标签的正弦波演示`

## 10. 前后端数据契约建议
为了让 UI 不再围绕内部 ID 拼接体验，建议后端补充以下字段。

### 10.1 任务列表项
推荐任务列表返回：

1. `task_id`
2. `display_title`
3. `status`
4. `updated_at`
5. `latest_summary`
6. `has_video`

### 10.2 任务结果
推荐任务结果返回：

1. `task_id`
2. `display_title`
3. `status`
4. `summary`
5. `video_download_url`
6. `preview_download_urls`
7. `validation_report_download_url`
8. `version_count`

### 10.3 最近视频聚合接口
为了避免前端先拉任务列表再逐条请求结果，建议新增聚合接口，例如：

`GET /api/videos/recent`

返回项建议至少包含：

1. `task_id`
2. `display_title`
3. `title_source`
4. `status`
5. `updated_at`
6. `latest_summary`
7. `latest_video_url`
8. `latest_preview_url`
9. `version_count`

第一版如果不想新增独立资源，也可以在 `/api/tasks` 中先补全这些轻量字段，但长期更推荐单独视频聚合接口。

## 11. 错误与空状态设计
中文优先不仅是标题中文，还包括错误和空状态可理解。

### 11.1 错误提示原则
错误提示应遵循：

1. 面向用户说明当前发生了什么。
2. 尽量给出下一步动作。
3. 不直接暴露 `http_401`、`http_500` 之类原始错误码作为主提示。

### 11.2 推荐文案方向
建议按场景给出中文提示：

1. 创建任务失败
   `任务创建失败，请检查描述或稍后重试。`
2. 视频未生成
   `当前版本还没有可播放的视频。`
3. 资源失效
   `视频链接已失效，请刷新任务结果。`
4. 会话过期
   `当前会话已过期，请重新登录。`

### 11.3 空状态
关键空状态需要明确设计：

1. 首页无视频
   `还没有可回看的视频`
2. 首页无任务
   `还没有任务`
3. 视频页无数据
   `最近没有符合条件的视频`
4. 详情页无结果
   `当前版本还没有结果摘要`

## 12. 响应式与移动端
中文优先页面不应只是桌面端可用。

### 12.1 桌面端
保持侧边导航和双栏结构，强调播放器、结果摘要和快捷修订。

### 12.2 移动端
推荐切换为底部导航：

1. `任务`
2. `视频`
3. `记忆`
4. `我的`

移动端首页优先显示最近视频卡片，再展示任务输入区，避免用户一上来只看到大表单。

### 12.3 响应式原则
移动端不隐藏关键功能，只调整展示顺序：

1. 视频播放器优先
2. 结果摘要其次
3. 修订框紧随其后
4. 资源与诊断信息折叠收纳

## 13. 视觉与文案原则
本轮设计建议保持现有的安静、克制、编辑式风格，但进一步适配中文阅读习惯。

### 13.1 中文排版
推荐调整：

1. 标题字距比英文版收紧。
2. 小标签不再大量依赖全大写逻辑。
3. 中文页面减少无意义的英文字段名。

### 13.2 内容优先级
视觉层级固定为：

1. 任务名称 / 视频名称
2. 视频预览与摘要
3. 状态和最近时间
4. 内部 ID

### 13.3 风格原则
继续保持：

1. 简洁
2. 现代
3. 易用
4. 不堆砌过度“运维感”组件

## 14. 测试策略
设计落地后，需要同步把测试基线切换到中文优先。

### 14.1 组件测试
覆盖以下内容：

1. 中文按钮文案
2. 中文标题
3. 中文空状态
4. 中文状态标签

### 14.2 页面流测试
建议覆盖主链路：

1. 创建任务
2. 最近视频出现
3. 打开任务详情
4. 提交中文修订

### 14.3 资源测试
重点验证：

1. `video_download_url` 可被网页播放器直接使用
2. 相对路径在配置 `VITE_API_BASE_URL` 场景下能正确解析
3. 预览图和视频资源在跨域部署下仍可访问

## 15. 分阶段实施建议
推荐拆成三个阶段上线，而不是一次性做完。

### Phase 1：中文化与视频回看基础
目标：

1. 全站中文优先
2. 首页增加最近视频区
3. 任务详情支持网页直接播放
4. 状态标签统一中文映射

### Phase 2：视频页与命名机制
目标：

1. 新增独立视频页
2. 后端补 `display_title`
3. 列表与详情页标题切换到展示名
4. 支持用户手动修改标题

### Phase 3：版本对比与更强迭代体验
目标：

1. 任务详情支持版本胶片带
2. 支持结果版本切换
3. 支持自然语言差异提示
4. 修订体验进一步产品化

## 16. 验收标准
当以下条件满足时，可认为这轮设计成功落地：

1. 中文用户打开首页后，能在 5 秒内看到最近视频并理解下一步操作。
2. 任务列表和视频页主标题不再默认显示 `task_id`。
3. 任务详情页可直接播放视频，并在同页完成中文修订。
4. 关键空状态和错误状态均为中文。
5. 记忆、画像、评测作为优化页存在，但不压过创作主链路。

## 17. 当前结论
本次设计已确认的最终方向如下：

1. 首页定义为“创作台”。
2. 结果展示以“最近视频”优先，而不是以任务 ID 列表优先。
3. 完成任务必须拥有贴近内容的展示名称。
4. 任务详情页升级为“视频工作台”。
5. 后续应补一个最近视频聚合接口和任务展示名字段。

在这个方向下，`easy-manim` 的 Web UI 将从“可用的工程控制台”进一步升级为“面向中文创作与运营场景的智能体视频工作台”。
